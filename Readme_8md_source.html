19 ## What
20 
21 [C++ Insights](https://cppinsights.io) is a [clang](https://clang.llvm.org)-based tool which does a source to source
22 transformation. Its goal is it to make things visible which normally, and intentionally, happen behind the scenes.
23 It's about the magic the compiler does for us to make things work.
24 
25 Take this piece of code for example:
26 
27 ```C++
28 class Base {
29 };
30 
31 class Derived : public Base {
32 };
33 
34 int main() {
35  Derived d;
36  Base& b = d;
37 }
38 ```
39 
40 Nothing special and of course it compiles. This is the compilers view on it:
41 
42 ```C++
43 class Base {
44 /* public: inline constexpr Base() noexcept; */
45 /* public: inline ~Base(); */
46 /* public: inline constexpr Base(const Base &); */
47 /* public: inline constexpr Base(Base &&); */
48 };
49 
50 class Derived : public Base {
51 /* public: inline constexpr Derived() noexcept; */
52 /* public: inline constexpr Derived(const Derived &); */
53 /* public: inline constexpr Derived(Derived &&); */
54 };
55 
56 int main(){
57  Derived d;
58  Base& b = static_cast<Base&>(d);
59 }
60 ```
61 
62 You can see all the compiler provided functions. Also the downcast from `Derived` to `Base`.
63 
64 I do not claim to get all the things right. This is just the initial version of insights I consider good enough to hand
65 it to the public. Keep also in mind that is solely based on clang and it's understanding of the AST.

67 ## Why
68 
69 [C++ Insights](https://cppinsights.io) is a [clang](https://clang.llvm.org)-based tool which does a source to source transformation.
70 Its goal is to make things visible which normally, and intentionally, happen behind the scenes. It's about the magic the compiler does
71 for us to make things work. Or looking through the classes of a compiler.
72 
73 Some time ago I started looking into some new things we got with C++11, C++14 and C++17. Amazing things like lambdas, range-based for-loops
74 and structured bindings. I put it together in a talk. You can find the [slides](https://www.andreasfertig.info/talks_dl/afertig-ndcolo-2017-fast-and-small.pdf)
75 and a [video](https://youtu.be/Bt7KzFxcbgc) online.
76 
77 However, all that research and some of my training and teaching got me start thinking how it would be, if we could see with the eyes of the
78 compiler. Sure, there is an AST-dump at least for clang. With tools like Compiler Explorer we can see what code the compiler generates
79 from a C++ source snippet. However, what we see is assembler. Neither the AST nor the Compiler Explorer output is in the language I write
80 code and therefore I'm most familiar with. Plus when teaching students C++ showing an AST and explaining that it is all there, was not
81 quite satisfying for me.
82 
83 I started to write a clang-based tool able to transform a range-based for-loop into the compiler-internal version. Then, I did the same
84 for structured bindings and lambdas. In the end, I ended up with doing a lot more as initially planned. It shows where operators are
85 invoked, places in which the compiler does some casting. C++ Insights is able to deduce the type behind `auto` or `decltype`. The goal
86 is to produce compilable code. However, this is not possible in all places.
87 
88 Still, there is work to do.
89 
90 I do not claim to get all the things right. This is just the initial version of [C++ Insights](https://cppinsights.io) I consider good
91 enough to hand it to the public. Also, keep in mind that it is solely based on clang and my understanding of C++ and the AST.
92

97 ## Building
98 
99 C++ Insights can be build inside the clang-source tree or outside.
100 
101 
102 ### Building outside clang
103 
104 You need to have a clang installation in the search path.
105 
106 ```
107 git clone https://github.com/andreasfertig/cppinsights.git
108 mkdir build && cd build
109 cmake -G"Ninja" ../cppinsights
110 ninja
111 ```
112 The resulting binary (insights) can be found in the build-folder.
113 
114 ### Building inside clang
115 
116 For building it inside the clang-source tree, assuming you have your source-tree already prepared:
117 
118 ```
119 cd llvm/tools/clang/tools/extra
120 git clone https://github.com/andreasfertig/cppinsights.git
121 
122 echo "add_subdirectory(cppinsights)" >> CMakeLists.txt
123 ```
124 
125 Then build clang as you normally do.
126 
127 ### cmake options
128 
129 There are a couple of options which can be enable with [!cmake](https://cmake.org):
130 
131 | Option | Description | Default |
132 |-------------------|:---------------------------| --------|
133 | INSIGHTS_STRIP | Strip insight after build | ON |
134 | INSIGHTS_STATIC | Use static linking | OFF |
135 | INSIGHTS_COVERAGE | Enable code coverage | OFF |
136 | DEBUG | Enable debug | OFF |

151 ## Usage
152 
153 Using C++ Insights is fairly simple:
154 
155 ```
156 insights <YOUR_CPP_FILE> -- -std=c++17
157 ```
158 
159 Things get complicated when it comes to the system include paths. There are path hard-coded in the binary which seem
160 to come from the compiler C++ Insights was build with. To help with that check out [scripts/getinclude.py](scripts/getinclude.py). It tries to
161 collect the system include paths from the compiler. Without an option it uses `g++`, you can also pass another compiler
162 as a first argument.
163 
164 Here is an example:
165 
166 ```
167 ./scripts/getinclude.py
168 -isystem/Applications/Xcode.app/Contents/Developer/Toolchains/XcodeDefault.xctoolchain/usr/bin/../include/c++/v1 -isystem/usr/local/include -isystem/Applications/Xcode.app/Contents/Developer/Toolchains/XcodeDefault.xctoolchain/usr/bin/../lib/clang/7.3.0/include -isystem/Applications/Xcode.app/Contents/Developer/Toolchains/XcodeDefault.xctoolchain/usr/include -isystem/usr/include
169 ```
170 
171 The script can be used together with C++ Insights:
172 
173 ```
174 insights <YOUR_CPP_FILE> -- -std=c++17 `./scripts/getinclude.py`
175 ```
