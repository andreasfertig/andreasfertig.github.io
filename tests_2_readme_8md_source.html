3 ## Running tests
4 Here you find files for regression tests. They can be invoked either directly with `runTest.py` or with a build
5 target:
6 
7 ```
8 cmake --build . --target tests
9 ```
10 
11 the later one tries to setup all the stuff required (compiler, include paths, path to the insights executable...).
12 
13 To run a single test you can use this:
14 ```
15 ./runTest.py --insights=PATH-TO-insights --cxx=PATH-TO-COMPILER TemplatesWithAutoAndLambdaTest.cpp
16 ```

18 ## What kind of tests
19 
20 In general this is a end-to-end verification system. There are no unit tests. There are only checks, if for a known input
21 a expected output is generated. C++ code level unit test would require create AST nodes. This is possible, but has still
22 the issue of beeing not as narrow as wanted. The end-to-end test seem best and worked out good so far.
23 
24 ## The test system
25 
26 The system is, that for each *.cpp file a corresponding *.expect file exists. This expect file contains the transformed
27 output. In some, hopefully rare, cases an additional file *.cerr exists as well. This file then contains the compiler
28 error output. The idea is to track all changes as close as possible.
29 
30 `runTest.py` does invoke insights as first and transforms the `*.cpp`. It then compares the result to the corresponding
31 `.expect` files contents. If there is a difference it shows the output of the diff. In any case it invokes gcc or clang
32 afterwards and tries to compile the transformed code.
33 
34 Currently, it is hard to do this tests. One reason is, that the transformation result is different between [libstdc++](https://gcc.gnu.org/onlinedocs/libstdc++/) and
35 [libc++](https://libcxx.llvm.org/). This results in many difference, if tests are executed under Linux vs. macOS. As my main development environment
36 is currently macOS all the test results are based on that.
37 
38 ## Test creation rules
39 
40 For each issue a dedicated `Issue#ISSUE_NUMBER.cpp` file is created with a matching `.expect` file. Both should be
41 checked in together with the actual fix.
42 
43 Other tests are named `NAME_OF_THE_TESTTest.cpp`. In case, there are more than one test for the same subject the scheme
44 is `NAME_OF_THE_TESTNUMBERTest.cpp`. For example:
45 
46 ```
47 StructuredBindingsHandlerTest.cpp
48 StructuredBindingsHandler2Test.cpp
49 ```

51 ## Updating tests
52 
53 There is a dedicated build target for cases where a change causes the need to update a lot of tests
54 
55 ```
56 make update-tests
57 ```
58 
59 Does update all failed tests as well as existing `.cerr` files. Be sure to check, whether the updated tests are in fact
60 correct.
